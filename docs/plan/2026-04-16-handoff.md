# Handoff — 2026-04-16

Revival pass on the dormant `arcpy_point_in_polygon_app`. Context: the
original was constrained to a single Windows tablet owned by the Office of
the Chief Judge (no JS SDK allowed, arcpy locked to that device). This pass
does two things: (1) get the code unstuck from that constraint, (2) sketch
what a future, backend-agnostic version would look like.

## What changed in this pass

### Migration: `arcpy` → ArcGIS API for Python (`arcgis`)

The core goal was to break the Windows-only dependency without rewriting
the service.

- **`geocoder.py`** — rewritten against `arcgis`:
  - `arcpy.geocoding.Locator` → `arcgis.geocoding.Geocoder` / `geocode()`
    against a `GIS` connection (AGOL anonymous by default; `locator_url` +
    authenticated `GIS` for Enterprise).
  - `arcpy.da.SearchCursor` → `pd.DataFrame.spatial.from_featureclass(...)`
    with `Geometry.contains()` for point-in-polygon. The local shapefile
    is still read from disk; only the iteration path changed.
  - `pathlib` for paths; `DEFAULT_SHAPEFILE` resolved relative to the
    module, not the CWD.
  - Shapefile `DataFrame` is lazy-loaded and cached, invalidated on
    `set_shape_file()`.

- **`geocoder_service.py`** — hygiene + correctness:
  - Dropped unused imports (`typing_extensions`, `jsonify`, `request`,
    `json`).
  - Fixed `'units\list_of_units.json'` (the `\l` was an invalid escape
    that worked by accident on Windows) → `os.path.join(...)` anchored to
    `__file__`.
  - Guarded `app.run()` under `if __name__ == "__main__":` and dropped
    `debug=True`; importing the module no longer starts a dev server.
  - `AssignByAddress` now returns structured `{supervisor, reason}`
    instead of silently returning `None` on miss.

- **`address_parser.py`** — removed unused `marshmallow` and the dead
  `resource` ctor arg; kept the reqparse behavior identical.

- **`unit_structure.py`** — dropped unused `Path`; fixed a malformed
  `list_of_units: []` annotation → `List[dict]`; made `get_supervisor`
  explicitly return `None`.

- **`requirements.txt`** — new. Pins `arcgis`, `flask`, `flask-restful`,
  `pandas`.

- **`README.MD`** — rewritten to describe the current (arcgis-based)
  stack, install/run, GIS auth, and the endpoint surface.

### Caveat flagged

The local `ArcGIS Files/Address_Points_geojson.loc` / `.loz` files are an
Esri-proprietary format only `arcpy` consumes. The new `geocoder.py`
reaches a **published** locator through a `GIS` connection. The legacy
`.loz` would need to be published to ArcGIS Online or Enterprise before
the service works against the original offline dataset. The
point-in-polygon side (the shapefile) still works locally unchanged.

## Upgrade path — planning direction

The revival works but inherits the vendor lock. The working idea for the
next phase is a **pluggable-provider architecture**, not a pluggable-GIS
architecture. The insight: only geocoding actually needs a GIS backend.
Point-in-polygon is a solved, vendor-neutral problem (`shapely` + STRtree)
and doesn't need Arc or QGIS at all.

### Proposed abstractions

```python
class GeocodingProvider(Protocol):
    def geocode(self, address: str) -> Point | None: ...

class DistrictResolver(Protocol):
    def district_for(self, point: Point) -> int | None: ...
```

### Implementations to ship

- **Geocoding providers**
  - `ArcGISOnlineProvider` — today's `geocoder.py` lifted behind the
    interface.
  - `ArcGISEnterpriseProvider` — published locator URL + auth.
  - `NominatimProvider` — open-source OSM geocoder, runs as a sibling
    container.
  - `LocalAddressPointsProvider` — fuzzy-match against the Cook County
    address points dataset. Arguably the best fit for this use case: no
    network, no license, deterministic.

- **District resolvers**
  - `ShapelyResolver` — load the shapefile once, build an STRtree index,
    do `.contains()` in pure Python. Fast, no GIS dependency.
  - `PostGISResolver` — `ST_Contains` against a PostGIS table, for the
    case where the district data ever needs to federate with other
    jurisdictions.

Selection is driven by config (env var or `config.yaml`) at startup. The
Flask layer doesn't know which backend is in use.

### Why not "QGIS or ArcGIS at install"

QGIS Server and ArcGIS Server containers are designed for WMS/WFS map
serving, not lookup services. Embedding one to answer
"which district is this point in?" is operational weight for no benefit.
If a policy requirement (chain-of-custody, audit) demands an ArcGIS chain,
that drives which **provider** is enabled, not which **GIS** is embedded.

## Potential next steps

Ordered roughly by dependency, not priority.

1. **Extract Protocols + move current code behind `ArcGISOnlineProvider`.**
   Half-day. No behavior change, just the seam.
2. **Ship `ShapelyResolver` with STRtree index.** Half-day. Biggest win —
   eliminates the GIS dependency for 99% of request time and makes the
   service trivially testable without any GIS install.
3. **Add `NominatimProvider` + `docker-compose.yml`** with an optional
   Nominatim container pre-loaded with Illinois data. One day. Gives the
   project a zero-cost, fully-offline geocoding option.
4. **Config-driven provider selection.** Env var or `config.yaml`. A few
   hours.
5. **Service Dockerfile.** Half-day.
6. **Tests.** Two days. Property-based tests for the district resolver
   (point inside → correct district; point outside → None); VCR/fixture
   tapes for each geocoding provider so tests don't hit live services.
7. **Resolve the district-overlap ambiguity.** District 2 appears under
   two supervisors in `list_of_units.json`. "First match wins in
   iteration order" is an accidental policy, not a real one. Needs a
   product decision — geography? calendar? explicit tiebreaker field?
8. **Auth on endpoints.** The service has no authentication. Fine for
   localhost, insufficient for a county-department deployment. JWT or
   mTLS depending on the deployment posture.
9. **Logging + metrics.** No observability today. At minimum: structured
   request logs, geocode/resolve latency histograms, miss-rate counters.
10. **Publish or retire the legacy locator.** Decide: either publish
    `Address_Points_geojson.loz` to AGOL/Enterprise for the
    `ArcGISEnterpriseProvider` path, or drop it in favor of the local
    address-points provider. Currently it's dead weight in the repo.
11. **Retire `templates/index.html`.** It's a cached Udemy stub calling a
    `/store` endpoint that never existed. Either build a real UI or
    remove it.
